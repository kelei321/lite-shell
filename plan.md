# LiteShell SFTP 修改计划

更新时间：2026-07-14  
状态：规划已建立，功能实施尚未开始  
执行原则：先修复数据安全和一致性问题，再完善传输可靠性，最后扩展交互功能。

## 1. 文档定位

本文件是 LiteShell SFTP 后续开发的唯一执行清单。

后续每个 SFTP PR 开始前必须先阅读本文件，并确认：

- 当前 PR 对应的阶段和任务。
- 前置 PR 是否已经合并。
- 本 PR 的范围、验收标准和测试要求。
- 是否需要用户进行真实服务器写入测试。

每完成一个 PR，必须在同一个 PR 中同步更新：

- `plan.md`：任务状态、实际完成内容、验证结果、遗留问题和下一步。
- `README.md`：面向使用者更新已完成能力、限制和使用说明。
- `handoff.md`：面向后续开发会话更新当前进度、关键实现、测试结果和下一任务。

禁止只修改代码而不更新以上三份文档。

## 2. 总体目标

SFTP 改造完成后应满足：

1. 多个 SSH 会话之间的目录、选择、历史和错误状态完全隔离。
2. 相同目标文件不会被并发任务同时写入。
3. 文件和目录冲突不会导致错误覆盖、移动或删除。
4. 每个传输任务都能可靠进入唯一终态，不会永久停留在运行中。
5. 断点续传能够验证源文件身份，不再仅依赖临时文件长度。
6. 递归传输不会跟随不受控符号链接、junction 或越界路径。
7. 文件冲突和目录冲突具有准确、可理解的语义。
8. 传输队列由后端统一管理，支持排队、暂停、恢复、取消和重启恢复。
9. SFTP 导航、批量操作、拖放和状态展示达到可日常使用水平。

## 3. 执行约束

### 3.1 PR 规则

- 每个 PR 只处理本计划中的一个任务。
- 必须从最新 `main` 创建分支。
- 不允许把无关 SSH、连接管理器、监控或 UI 重构混入 SFTP PR。
- 禁止整文件无目的重写，优先提取小型模块并保留现有业务行为。
- PR 创建前必须比较 `main...branch`，确认只有目标文件。
- PR 描述必须包含 `Summary`、`Validation`、`Scope`。
- 必须等待 CI 通过并完成 code review 后才能合并。
- 默认使用 squash merge。

### 3.2 验证规则

常规验证：

```powershell
npm run validate
```

涉及桌面交互时还需要：

```powershell
npm run desktop
```

`npm run desktop:build` 属于完整桌面构建，只有在发布或安装包验证阶段执行。

### 3.3 服务器安全边界

未经用户重新明确授权，不得在用户服务器上执行：

- 上传文件。
- 创建目录。
- 重命名。
- 删除。
- 写文件。
- 修改权限。

开发和自动测试优先使用：

- Rust 单元测试和临时目录。
- 前端纯状态测试。
- 本地可控测试服务。
- 用户自行准备的专用临时服务器目录。

### 3.4 文档状态标记

任务状态只能使用：

- `待开始`
- `进行中`

- `已完成`
- `阻塞`

只有代码已经合并到 `main`，并且要求的 CI 和测试通过后，才能标记为 `已完成`。

## 5. PR1：SFTP 会话状态隔离

状态：`待开始`

### 目标

解决旧会话异步目录请求覆盖新会话、跨服务器保留错误选中项，以及使用错误会话执行删除、重命名或下载的问题。

### 当前问题

当前页面共用一套：

- 路径。
- 文件列表。
- 加载状态。
- 错误信息。
- 选中项。
- 前进后退历史。

`loadDirectory` 发起异步请求后直接写入全局状态，没有验证请求所属会话和请求版本。快速切换标签时，服务器 A 的旧请求可能覆盖服务器 B 的界面。

### 实现范围

- 定义 `SftpSessionState`，至少包含：
  - `sessionId`
  - `path`
  - `entries`
  - `loading`
  - `error`
  - `selectedEntries`
  - `history`
  - `historyIndex`
  - `requestVersion`
- 使用 `Map<sessionId, SftpSessionState>` 保存每个会话状态。
- `loadDirectory` 显式接收 `sessionId`，不能在请求完成时重新读取活动会话。
- 每次加载递增该会话的 `requestVersion`。
- 只有请求仍为该会话最新请求时才允许写回列表、错误和 loading。
- 删除、重命名、下载使用选中项所属会话，不使用可能已变化的 `activeSessionId`。
- 切换标签时恢复各自路径和历史。
- 关闭会话时清理状态、选择和待处理请求标记。
- 失败请求不得清空另一个会话的成功列表。

### 推荐模块

```text
src/sftp/session-state.ts
src/sftp/session-state.test.ts
```

`App.vue` 只负责绑定 UI 和调用状态模块，避免继续扩大单文件复杂度。

### 测试

- A 请求延迟时切换到 B，A 结果不能覆盖 B。
- 同一会话连续请求路径 1 和路径 2，路径 1 后返回时不能覆盖路径 2。
- 旧请求的 `finally` 不能关闭新请求的 loading。
- A 的选中项不能在 B 会话中触发删除、重命名或下载。
- 会话关闭后迟到响应被忽略。
- 书签和历史仍按 profile/session 隔离。

### 验收标准

- 两个服务器快速切换时，各自路径、列表、选择和历史保持正确。
- 所有远程写操作都能证明其 `sessionId` 与选中项归属一致。
- 新增前端单元测试覆盖核心竞态。
- CI 全部通过。

### 完成记录

## 6. PR2：相同目标路径传输互斥

状态：`待开始`

### 目标

避免多个上传或下载任务同时写入同一个目标文件及其临时文件。

### 当前问题

后端允许最多 3 个传输任务并发，但临时文件使用固定名称：

```text
{target}.liteshell.part
```

相同目标的并发任务可能同时打开、截断、追加、提交或删除同一个临时文件。

### 实现范围

- 在 `SftpTransferManager` 中增加目标路径锁。
- 定义稳定的 `TransferTargetKey`：
  - 会话或服务器标识。
  - 方向。
  - 规范化目标路径。
- 同一目标只允许一个活动任务。
- 不同目标仍保持最多 3 路并发。
- 冲突返回：

```text
TRANSFER_TARGET_BUSY
```

- 成功、失败、取消、panic-safe 清理路径都必须释放锁。
- 本地 Windows 路径比较处理大小写和规范化。
- 远程路径尽量使用 canonical path；无法 canonicalize 的新路径使用规范化父路径和文件名。

### 测试

- 相同远程目标的两个上传任务互斥。
- 相同本地目标的两个下载任务互斥。
- 不同目标可以并行。
- 任务失败或取消后锁会释放。
- 排队任务不应提前永久占有目标锁。

### 验收标准

- 不存在两个任务同时写同一个 `.part`。
- UI 对重复任务显示明确中文提示。
- 原有全局并发限制继续生效。

## 8. PR4：统一传输终态和清理

状态：`待开始`

### 目标

保证每个任务最终且只会进入一个终态，所有资源和临时状态都能清理。

### 当前问题

数据复制完成后还有：

- flush/shutdown。
- 备份原目标。
- 提交临时文件。
- 删除备份。
- 关闭 SFTP。
- 清除取消标记。

部分步骤直接返回错误，可能没有发送 `failed` 事件，导致 UI 永久停留在 `running`。

### 实现范围

抽取统一传输事务或 finalizer，负责：

- 初始 `running` 事件。
- 数据复制。
- flush/shutdown。
- 原目标备份。
- 临时文件提交。
- 失败回滚。
- SFTP 关闭。
- 取消标记清理。
- 目标锁释放。
- 唯一终态事件。

终态：

```text
completed
failed
cancelled
```

为后续暂停功能预留：

```text
paused
```

清理策略：

- 成功：删除临时文件、备份和检查点。
- 可续传失败：保留经过验证的临时文件。
- 源变化或临时文件损坏：删除或隔离临时文件。
- 提交失败：优先恢复原目标。
- 用户取消：按明确策略保留或删除断点。

### 测试

- shutdown 失败。
- 原目标备份失败。
- 临时文件提交失败。
- 回滚成功和回滚失败。
- 取消发生在读取、写入和提交阶段。
- 每个任务最多一个终态事件。

### 验收标准

- UI 不会遗留永久运行任务。
- 失败后锁和取消标记均释放。
- 原目标能够恢复时必须恢复。
- 不遗留无主 backup 文件。

## 11. PR7：明确目录冲突语义

状态：`待开始`

### 目标

将当前目录“覆盖”实际执行的合并行为改成准确语义，并增加安全的目录替换方案。

### 文件冲突

继续支持：

```text
覆盖
跳过
重命名
```

### 目录冲突

改为：

```text
合并
跳过
重命名
替换
```

含义：

- 合并：保留目标中源目录不存在的内容。
- 跳过：整个目录不处理。
- 重命名：创建新的目标目录。
- 替换：安全备份旧目录后复制新目录。

### 实现范围

- 目录默认策略为“合并”。
- “替换”必须二次确认。
- 文件和目录的“应用于全部”策略分开保存。
- 服务器不支持安全目录 rename 时，不得静默递归删除目标目录。
- UI 明确提示替换会删除目标中的额外内容。

### 测试

- 合并保留目标额外文件。
- 跳过不创建任何内容。
- 重命名生成稳定新名称。
- 替换失败时恢复原目录。
- 不支持安全替换时返回明确错误。

### 验收标准

- UI 文案与实际行为一致。
- “覆盖目录”不再被误用来表示合并。

- 每会话独立前进、后退和当前路径。
